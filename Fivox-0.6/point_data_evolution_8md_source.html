1 Evolution of data at a point over time {#pointDataEvolution}
2 ======================================
3 
4 One of the features that would be extremely useful for scientists is the
5 ability to obtain the evolution of the data at a specific point over time.
6 For example, they could load a current report and compute the LFP for a specific
7 target, and once they have their full volume, they could focus on a specific
8 point of the scene (e.g. a voxel), and obtain a 2D curve showing the evolution
9 of voltage at that point for the whole available time span.
10 
11 ## Requirements
12 
13 * The new feature will be provided as a separate command line tool.
14 * Users should be able to specify the coordinates of a 3D point as input.
15 * As output, users will obtain a series of values corresponding to the evolution
16  of the data at that specific point over time, one value per simulation frame,
17  using always the default report dt.
18 * The computation should be independent from the voxelization step, as the
19  point for which the values will be computed should not be limited to discrete
20  voxels, but a continuous 3D space.
21 * This feature is not related to the (online) streaming of values. It is just an
22  additional (offline) step that is done after loading and processing the data,
23  to report the information in a more localized way.
24 
25 ## Implementation
26 
27 The core of the computation is already present in Fivox in the form of functors,
28 that take a 3D point as a parameter and, based on the use case and the different
29 types of functors available, return a value that is the result of evaluating the
30 existing events.
31 
32 To make the computation of these values easy from a user point of view, we need
33 to expose the usage of the functors in a separate step as an independent
34 application.
35 
36 For that reason, the 'voxelize' app will need to be refactored to extract the
37 common parts into a superclass that will be used as base for all the
38 applications.

61 ## Usage
62 
63 The usage of the C++ app would be as:
64 
65  sample-point --volume "fivox://BlueConfig?target=Column&dt=0.1" --point [x, y, z] --frame[s] "0 10" [or alternatively, --time[s] "0 100"] --output data.txt
66 
67 And the plotting tool:
68 
69  plot2D.py --input data.txt --output graph.png
70 
71 If the output file name is not specified, an interactive matplotlib widget will
72 show, allowing the user to navigate through the graph and explore the different
73 values.
74 
75 Potentially, both steps could be combined into one wrapping tool that runs both
76 steps one after the other.
77 
78 ## File format
79 
80 The resulting plot should reflect the time evolution on the X axis, and the data
81 values on Y. Therefore, the C++ tool will need to generate files in the
82 following format:
83 
84 * One line per timestep
85 * Each of the lines containing the timestamp (ms) and value (output unit),
86  separated by whitespaces
87 * An optional header containing the information about the way the data was
88  generated (Fivox volume URI, dt, frame range and point coordinates)
89 
90 Example:
91  # File generated by the sample-point tool:
92  # - Format: timestamp value
93  # - Fivox URI: fivox://BlueConfig?report=soma&functor=lfp
94  # - dt: 0.1
95  # - Frame range: [ 0 10 ]
96  # - Point sampled: [ 0 0 0 ]
97 
98  0 -0.631731
99  0.1 -0.631102
100  0.2 -0.630923
101  0.3 -0.630797
102  0.4 -0.630697
103  0.5 -0.630615
104  0.6 -0.630545
105  0.7 -0.630486
106  0.8 -0.630434
107  0.9 -0.63039

109 ## Issues
110 
111 ### 1: Why should we create a new application?
112 
113 The alternative is to overload the 'voxelize' app. Since we don't want to depend
114 on the voxelization step or the volume creation for the computation of the
115 values, we will create a new application that will have some parts in common
116 with other apps, such as the volume URI evaluation or the frame/time ranges to
117 compute.
118 
119 As for the name of the new application, here are some suggestions:
120 values-over-time, values-at-point, data-over-time, data-evolution, sample-point,
121 extract-timeseries... or any combination of the aforementioned.
122 
123 ### 2: How to output the resulting values?
124 
125 There are several possibilities:
126 
127 1. Writing to file
128 2. Plot directly as a 2D graph
129 3. Stream through ZeroEQ
130 
131 Option 2 would require the integration of a plotting library in C++. Option 3
132 adds complexity that is not needed now, as it would only make sense for
133 the streaming case, which is not a priority at the moment.
134 
135 Resolved: For the sake of simplicity, we will start with a basic use case in
136 which we write from the C++ app to a file and then add a separate step to create
137 the 2D plot using a simple library in Python (matplotlib).
138 
139 ### 3: File format: binary or ASCII?
140 
141 Resolved: For readability, and since the file structure is simple enough and
142 there are not many potential bottlenecks in read/write performance, the format
143 will be ASCII.
