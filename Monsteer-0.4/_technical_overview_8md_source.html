6 # Technical Overview {#Technical_Overview}
7 
8 The Monsteer library helps users setting up loosely coupled
9 simulation-observer/controller pairs in an easy manner. After the
10 integration of all the components, users can interact with a simulation both
11 for streaming events from it and steering any of its parameters.
12 
13 To accomplish this task, Monsteer provides: Brion plugins for spike data streaming,
14 a Python interface to steer the simulator and a bridging application between
15 the user and the simulator.
16 
17 At the moment, Monsteer only offers support for the \ref NEST simulator which is
18 mainly used by the Neurobotics group in the BlueBrain project. Despite this
19 limitation, all the Monsteer components were developed thinking on their
20 reusability with other simulators such as Neuron.
21 
22 For example, Monsteer is based on the plugin mechanism provided by Brion, where the
23 interface of spike streaming is defined. By using different schemes,
24 developers have the ability of implementing different spike
25 readers/writers. These are all interfaced through the BBPSDK API, where the
26 exposed interface is common for all the spike stream processes. As a result,
27 for users it is transparent to use different spike sources where they only
28 have to provide different spike readers or writers. These mechanisms are
29 already used to stream and read file data from NEST and Neuron simulators by
30 just using different schemes.
31

36 # NEST Neural Network Simulator {#NEST_Simulator}
37 
38 NEST is a large-scale neural network simulator that can be run on distributed
39 computing environments. The data flow in the distributed running environment is
40 mainly based on the MPI library. Different NEST processes can either share data
41 either by just using MPI, or through the \ref Music_Library, a library that
42 facilitates the unification of different simulators using a common interface.
43 
44 Monsteer uses the MUSIC interface integrated in the NEST simulator for steering and
45 streaming. The functionalities are exposed in the Python interface of the
46 simulator. This interface allows the Python classes, namely \ref Nesteer, to be
47 developed in order to provide support for sending and receiving steering
48 messages and spike events.
49 
50 # MUSIC Library {#Music_Library}
51 
52 MUSIC is a well-known data exchange library for neural simulators such as
53 NEST, Neuron and Moose. It can connect large scale neural networks and
54 has an easy interface for coupling applications with the simulators.
55 
56 Using static configurations between applications which are running at
57 different speeds, the MUSIC library is capable of synchronizing data by
58 employing internal buffering and synchronization mechanisms. Developers using
59 MUSIC to build such applications are abstracted from this synchronization
60 mechanisms, making the development process easier rather than doing it by
61 low-level MPI calls.
62 
63 MUSIC is used in the Music Proxy application to receive spikes from a NEST
64 simulator and to communicate the steering messages to and from the
65 simulator. The underlying mechanisms in the library helped the proxy
66 application to be developed in an very flexible way, also making it easily
67 extensible to make the coupling with other libraries.
68 
69 # Brion Library {#Brion_Library}
70 
71 Brion provides many additional plugins and plugin interfaces for accessing
72 BlueBrain project data. This data includes neuron structures, neuron
73 placements, simulation results, meshes, etc. The plugins in this library are
74 activated by a scheme, where a scheme can also be a file extension rather than
75 a fully qualified URL.
76 
77 Monsteer provides a standalone spike reader and writer plugin for Brion. This
78 plugin is automatically registered by Brion at runtime and can be instanciated
79 by passing a "monsteer://" URI to the brion::SpikeReport constructor.
80 
81 # Brion Spikes {#Brion_Spikes}
82 
83 Spike events are one of the main simulation results where visualization and
84 analysis is important for neuroscientists. A spike event is a binary event
85 which takes place in a neuron on a certain time within the simulation, when
86 neuron passes a certain voltage threshold. This spike event can be represented
87 as a timestamp and a neuron identifier. In Brion and Monsteer, there is support
88 for internal BBP data formats, NEST file formats and streaming formats.
89 
90 # Zeq ZeroEq: Auto-discovery and Event Based Data Sharing Library
91 
92 The ZeroEQ library is used for auto-discovery of services, data transport and
93 event based messaging. The auto-discovery and data transport are done
94 through the ZeroMQ library, where in lack of auto-discovery, it is also possible
95 to target a host with a given port. The messages are based on FLATBuffers scheme
96 which enforces the messages to be well-defined. On top of these libraries,
97 ZeroEQ provides an URI-based auto-discovery, event listening and event
98 registration layers. In ZeroEQ the events form a vocabulary for a given scheme
99 and can easily be extended.
100 
101 Two ends of the connection in ZeroEQ is a publisher and subscriber pair. When
102 auto-discovery is possible, a publisher announces its scheme to the outer world
103 in the network and subscribers interested with this scheme can connect to this
104 publisher.
105 
106 When auto-discovery is not available or is not requested by the subscriber, it
107 is also possible to provide a specific host and port in the scheme. This way,
108 a connection to a known publisher can be established.
109 
110 In Monsteer, ZeroEQ is used for: sending spike events from Music Proxy to the
111 steering application, sending steering messages from the steering application
112 and sending information messages from the simulator. Each item introduces a
113 scheme, its own vocabulary and consequently its own events.
114 
115 # Music Proxy {#Music_Proxy}
116 
117 The Music Proxy acts as a bridge between the simulation and the user
118 application, making use of MUSIC communicate steering messages between both
119 components, as well as receiving spike events from the simulation in order to
120 send them later to the user application.
121 
122 The application is composed of two main components: the MUSIC spike event
123 handler and the MUSIC steering handler. Within the application, these
124 components are synchronized on the same loop to send and receive data.
125

127 below.

131 ## Spike Event Handler {#Music_Proxy_Spike_Event_Handler}
132 
133 The spike event handler is responsible for receiving the spike events and
134 forwarding them to the application using the ZeroEQ library. MUSIC has
135 mechanisms to register for spike events and using the synchronization
136 mechanisms available in the library, these events are captured and translated
137 into the spike data format used in Brion.
138 
139 ## Spike Streaming {#Spike_Streaming}
140 
141 The spike event reader and writer interfaces are declared in Brion. The
142 necessary structures for reading/writing files and streaming spikes are
143 developed within the Brion and Monsteer libraries.
144 
145 Monsteer provides a spike streaming plugin based on the ZeroEQ library to
146 transfer events from one end to another. Both endpoints, the publisher and
147 the subscriber, can generally rely on ZeroEQ's auto-discovery mechanism to
148 automatically connect to each other. Otherwise, it is also possible to pass a
149 specific host and port to the brion::SpikeReport plugin constructor. Please
150 refer to the ZeroEQ documentation if you want to learn more about the
151 connection mechanism and its filtering options.
152 
153 ## Steering Handler {#Music_Proxy_Steering_Handler}
154 
155 The steering handling is based on two main components. The first component
156 forwards steering messages from the steering application to the NEST
157 simulation. The receipt of steering messages is handled through the Zeq
158 library and the forwarded messages are sent through the MUSIC library.
159 
160 The second component receives messages from the NEST simulation and forwards
161 them to the steering application. Messages are received from the NEST
162 simulation through the MUSIC library, and sent to the user application through
163 ZeroEQ.
164 
165 # BBPSDK Simulation Controller {#BBPSDK_Simulation}
166 
167 BBPSDK is a basic library for accessing the BlueBrain data. It can read spike
168 reports, compartment reports, meshes, neuron structures, their placements,
169 etc. It provides a complete user API and their Python wrappings where the
170 implementations are generally plugins. The right plugin is selected based on a
171 given scheme. These plugins are generally located in Brion library and can
172 easily be extended by 3rd party libraries. Monsteer library is one of the libraries
173 that provides plugins for the Monsteer scheme.
174 
175 For users to steer a simulation easily, a new functionality was added in the
176 BBPSDK library. The functionality is abstracted from the underlying simulator
177 using the plugin mechanism. At the moment there is only one plugin in Monsteer for
178 steering NEST simulations, but for the future new functionalities can be added
179 for different simulators such as Neuron. With the simulator interface, a user
180 can inject stimuli into a running simulation, play and pause the simulation
181 and receive additional information from it.
182 
183 The functionality added in BBPSDK was also wrapped in Python to make it more
184 useful for applications that have a Python interface i.e: RTNeuron. Later,
185 the Python wrapping will be extended for the NEST use case in monsteer.
186

192 # Nesteer: A Python interface for processing the steering messages {#Nesteer}
193 
194 NEST users generally use a Python shell for constructing and running their
195 simulations. If at some point they want to add steering and streaming
196 capabilities to their simulation, Monsteer can facilitate this process by providing
197 a Python object to process the received messages. Using the provided Python
198 object, a user can easily define what to do when a steering message is
199 received by providing new functions for certain events.
200 
201 Nesteer provides internal mechanisms to setup the MUSIC communication channels
202 and connect the neurons. For its use with BBP data, it provides a mapping
203 between NEST neuron ids and BBP circuit neuron ids. That way, users can
204 transparently work with BBP circuits.
