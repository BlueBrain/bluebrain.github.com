6 # Introduction {#Introduction}
7 
8 This page is a startup guide for any C++ developer willing to make its
9 application available on the web. A step by step tutorial illustrates the
10 definition and implementation of application specific features that will easily
11 be accessed through a REST interface.

15 ## HeartBeat event {#HeartBeatEvent}
16 
17 The heart-beat event is sent on a regular bases to inform the RESTBridge that
18 the rendering resource is running and ready to process requests. When the
19 RESTBridge receives the first heart-beat event from the rendering resource, the
20 vocabulary available in the rendering resource is requested and made available
21 to the user-end web application. The RESTBridge will keep on returning 501 HTTP
22 errors (Resource not available) to the user-end web application until the
23 rendering resource vocabulary is registered.
24 
25 ~~~
26  #include <zeq/zeq.h>
27  #include <zeq/hbp/hbp.h>
28  #define DEFAULT_HEARTBEAT_TIME 1000.0f
29  lunchbox::Clock heartbeatClock;
30  ...
31  // In this example, the rendering resource is constantly rendering,
32  // and the following code is executed for every frame.
33  if( heartbeatClock.getTimef() >= DEFAULT_HEARTBEAT_TIME )
34  {
35  heartbeatClock.reset();
36  publisher.publish( zeq::Event( zeq::vocabulary::EVENT_HEARTBEAT ) );
37  }
38 ~~~

40 ## Vocabulary event {#VocabularyEvent}
41 
42 The vocabulary event is used to register the commands that are supported by the
43 rendering resource. The rendering resource can publish this event at any time to
44 inform the RESTBridge of any changes in its functionalities. There are two
45 types of events:
46  - Publisher: An event that is emitted by the rendering resource.
47  - Subscriber: An event that is received by the rendering resource.
48 
49 As an example, a ChangeBackgroundColor event would be a subscriber, and an
50 ImageJPEG event a publisher.
51 If an event can be both a subscriber and a publisher, then it should be
52 registered twice.
53 
54 A vocabulary event contains the entire list of supported events and completely
55 replaces any existing registration. In other words, any event that is not listed
56 in the vocabulary is removed from the RESTBridge internal maps, even it was
57 previously registered by another vocabulary event.
58 The following code snippet illustrates the use of the vocabulary event to
59 register ImageJPEG and Camera events.
60 
61 ~~~
62  #include <zeq/zeq.h>
63  #include <zeq/hbp/hbp.h>
64  // Declare vocabulary
65  zeq::EventDescriptors vocabulary;
66  // Add ImageJPEG event to vocabulary
67  vocabulary.push_back( zeq::EventDescriptor( zeq::hbp::IMAGEJPEG,
68  zeq::hbp::EVENT_IMAGEJPEG,
69  zeq::hbp::SCHEMA_IMAGEJPEG,
70  zeq::PUBLISHER ) );
71  // Add CAMERA event to vocabulary
72  vocabulary.push_back( zeq::EventDescriptor( zeq::hbp::CAMERA,
73  zeq::hbp::EVENT_CAMERA,
74  zeq::hbp::SCHEMA_CAMERA,
75  zeq::SUBSCRIBER ) );
76  // Publish vocabulary to RESTBridge
77  const zeq::Event& vocEvent =
78  zeq::vocabulary::serializeVocabulary( vocabulary );
79  publisher.publish( vocEvent );
80 ~~~

82 ## Camera event {#CameraEvent}
83 
84 The camera event is defined by a 4x4 matrix, with values defined in microns. The
85 FlatBuffers definition of the event is the following:
86 
87 ~~~
88  namespace zeq.hbp;
89  table Camera
90  {
91  matrix:[float];
92  }
93  root_type Camera;
94 ~~~

96 ## ImageJPEG event {#ImageJpegEvent}
97 
98 The ImageJPEG event is defined by an array of bytes containing a 32bit BGRA
99 image. The FlatBuffers definition of the event is the following:
100 
101 ~~~
102  namespace zeq.hbp;
103  table ImageJPEG
104  {
105  data:[ubyte];
106  }
107  root_type ImageJPEG;
108 ~~~
109 
110 Here is a simple example of how to generate and publish the event in the
111 rendering resource application code:
112 
113 ~~~
114  #include <zeq/zeq.h>
115  #include <zeq/hbp/hbp.h>
116  const uint8_t* dataPtr = ...; // Buffer containing the JPEG image bytes
117  const uint64_t dataSize = ...; // Buffer size
118  const zeq::hbp::data::ImageJPEG image( dataSize, dataPtr );
119  const zeq::Event& image_event = zeq::hbp::serializeImageJPEG( image );
120  publisher.publish( image_event );
121 ~~~

123 ## ZeroEQ Handlers {#ZeroEQHandlers}
124 
125 For every ZeroEQ event supported by the C++ application, a handler must be
126 defined and registered. A Handler is a class method that implements the code
127 related to the ZeroEQ event. The following example illustrates the registering
128 of the MyApplication::onCamera method that takes care of the ZeroEQ Camera
129 event.
130 
131 ~~~
132  subscriber_ = new zeq::Subscriber(servus::URI( zeqSchema_ + "cmd://" ));
133  subscriber_->registerHandler( zeq::hbp::EVENT_CAMERA,
134  boost::bind( &MyApplication::onCamera, this , _1 ));
135 ~~~

137 ## RESTBridge Server {#RESTBridgeServer}
138 
139 The RestBridge class implements the http server and starts a dedicated thread on
140 the host and port specified in the class constructor. HTTP requests addressed to
141 the HTTP server is transformed into a ZeroEQ event, according to the registered
142 vocabulary. Only GET requests expect a ZeroEQ event in return. If the HTTP
143 request contains a unknown command (not registered in the vocabulary), a
144 BAD_REQUEST (HTTP 400) error is returned.
145 
146 ~~~
147  restbridge::RestBridge mainServer("localhost", 3000);
148  mainServer.run("restSchema");
149 ~~~
