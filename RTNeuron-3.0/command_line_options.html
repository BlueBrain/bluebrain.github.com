Command line options

General options
 --demo [demo_name] Starts the demo of the given name. If no name is provided it will start the default demo (named projections). This demo shows a default circuit using somas. When a soma is clicked, the anterograde pathways of the neuron are shown, if the soma is clicked again, the retrograde pathways are shown. The shell will also be started.
 --demo help Print out the list of available demos.
 --gui Launch the GUI. This option is not compatible with --eq-config. It can be combined with --shell.
 --shell Start an interactive IPython shell after initialization (or a regular console if IPython is not available). This option can be combined with --gui.
 --version Displays version information.
 -h --help Displays the command line options help.
Data loading options
 --neuron -n gid [mode [color]] The MVD file id (starting at 1) of a neuron to be loaded and the its display options. Mode is one of soma|detailed|no_axon|skeleton|none and color is one o RGB[A]|random|all-random|by-branch [RGBA RGBA]|by-width[@attenuation] [RGB[A]]|by-distance-to-soma [RGB[A]].
 --neurons gid gid [mode [color]] The MVD file id range for a list of neurons to be loaded and their display options. Mode is one of soma|detailed|no_axon|skeleton|none and color is an RGB[A] tuple or one of RGB[A]|random|all-random|by-branch [RGBA RGBA]|by-width[@attenuation] [RGB[A]]|by-distance-to-soma [RGB[A]].
 --target targetname [mode [color]] The name of a target to load and its display options. Mode is one of soma|detailed|no_axon|skeleton|none and color is one of RGB[A]|random|all-random|by-branch [RGBA RGBA]|by-width[@attenuation] [RGB[A]]|by-distance-to-soma [RGB[A]].
 -b --blue-config filename The simulation configuration file to use.
Style options
 --accurate-headlight Accurate rendering of headlight lighting (required if head tracking is used).
 --afferent-syn-color red green blue alpha Default color for the glyph representing the afferent position of synapses. Applied to synapse targets appearing after this option in the command line.
 --background red green blue [alpha] Background RGBA color. Alpha channel defaults to 1 if not given. Frame grabbing considers the alpha channel of the background. If alpha equals to 1, the output image will have no alpha channel.
 --efferent-syn-color red green blue alpha Default color for the glyph representing the efferent position of synapses. Applied to synapse targets appearing after this option in the command line.
 --electron Render neurons with a shader that mimics electron microscope imaging (SEM)
 --idle-AA Enable idle anti-aliasing (16 samples per pixel by default)
 --neuron-color red green blue Default overall color for neurons when no simulation data is being displayed. Applied to neuron targets appearing after this option in the command line.
 --neuron-color all-random Use a random opaque color per neuron by default. Applied to neuron targets appearing after this option in the command line.
 --neuron-color alpha‑by‑width[@attenuation] [R G B [A]] Use variable color per vertex depending on the neuron branch width. The final color is the interpolation between the optional RGBA base color and the color ((R + 0.5, G + 0.5, B + 0.5, 1) using 1 - exp(-width × 1/attenuation) as the interpolation factor. If the base color is not given it defaults to (0, 0.5, 1, 1). If the optional attenuation parameter is not given it defaults to 2. As can be seen, the base color is the color applied to the thinnest branches.
 --neuron-color by-branch Color axons with blue and dendrites and soma with red. This option overrides any other coloring scheme for structural display of neurons (i.e. without simulation data). Applied to neuron targets appearing after this option in the command line.
 --neuron-color random Initialize the default color for neurons with a random color. Applied to neuron targets appearing after this option in the command line.
 --soma-radius microns If present, specifies the default soma radius to use when the display mode is soma and the morphology is not available (e.g. --no-morphologies was present).
Simulation Playback
 --show-spikes Deprecated, use --spikes instead.
 --sim-step Sets the simulation playback step. When no interface is shown this step is measured in milliseconds.
 --spike-tail length The length (in time) for spike trails along the axon and the decay time for spikes in the soma.
 -r --report name The report name to load data from. This name must be present in the Blue Config file passed with -b.
 -s --spikes [spike_url] Load and show the spikes from the SpikePath pointed by the blue config. A different spike URL can be provided as an optional argument.
 -w --sim-window start_time stop_time The time window of interest for simulation playback (times in ms). If compartment or spikes reports are loaded, the final interval is the intersection between this window and the reports' windows.
Scene management options
 --mesh-partition Use meshes and not only morphologies to compute the distribution of geometry complexity used to choose the cut plane positions of spatial DB.
 --no-cull Do not perform fine grained culling of neuronal. The only test left is a simple per neuron axis-aligned bounding-box intersection test.
 --no-strips Do not use triangle strips as render primitives.
 --round-robin-DB-partition Use heuristic round-robin partition of the neuron targets for DB decomposition modes. Each instance will only load the meshes and morphologies it needs.
 --spatial-DB-partition Use a balanced k-d tree based on the morphological points of the neurons to divide the scene in DB decompositions. Each instance will load permanently only the meshes and morphologies intersected in its k-d tree leaf.
 --unique-morphologies Assume that neuron morphologies are unique for scene building. Off by default, but RTNeuron will try to determine automatically if morphologies are unique using the morphology names. Assuming unique morphologies enables lazy loading of meshes.
Levels of detail
 --clod Use tubelets and pseudo cylinders levels of detail with a continuous transition between both depending on branch thickness and distance to the camera. Meshes are used for the cell cores unless --no-meshes is provided.
 --no-lod Do not generate different levels of detail. Only applicable to neurons displayed with the detailed and no_axon modes.
 --no-meshes Do not load mesh data.
 --no-morphologies Do not load morphologies if possible. This applies to scene configurations in which only neuron targets with display mode soma are used. Using any other display mode causes this option to have no effect.
 --use-tubelets Use tubelets instead of pseudo cylinder for some levels of detail.
Network options
 --sync-camera zeroeq-session Synchronize the camera with other applications.
 --sync-selections zeroeq-session Synchronize selections with other applications. When --sync-selections is used this option tells RTNeuron to react to toggle selection request events and track the selection state of neuron internally. Off by default, this means that RTNeuron simply sends toggle events and waits for an external selection event to update its selections.
 --zeroeq-http-server [host[:port]] Enable the REST interface. All command line options are passed to the zeroeq::http::Server object for its initialization.
GUI options
 --tracker device@host[:port][&sensorid=(any|###)] A VPRN device to be used as head tracker. Only available if compiled with VRPN support.
 --use-spacemouse device@host A VRPN spacemouse device for camera manipulation. Only available if compiled with VRPN support. Deprecated, use --vrpn-manipulation instead
 --use-wiimote device@host A VRPN Wiimote device for camera manipulation. Only available if compiled with VRPN support and Wiimote support. Deprecated, use --vrpn-manipulation instead.
 --use-wiimote-pointer device@host A VRPN Wiimote device to use as 3D pointer. Only available if compiled with VRPN support and Wiimote support.
 --vrpn-manipulator DeviceConfig Use VPRN devices as a camera manipulator. Only available if compiled with VRPN support. The option argument specifies the device configuration as either a JSON string or a path to a JSON file.
  • DeviceConfig: Either a DevConfigStr or DevConfigFile.
  • DevConfigStr: Valid JSON devices configuration string.
  • DevConfigFile: Valid JSON devices configuration file.
A valid JSON device configuration has the following format: 
 {
   "type": DeviceTypeStr,
   "tracker":
   {
     "url": DeviceURL,
     "sensorid": Id | "any",
     "position_axis": AxisSpec | Matrix3f,
     "attitude_axis": AxisSpec | Matrix3f
   },
   "button": DeviceURL | { "url" : DeviceURL }
   "analog": DeviceURL | { "url" : DeviceURL }
   "url": DeviceURL
 }
Where:
  • DeviceTypeStr: one of "GYRATION_MOUSE", "INTERSENSE_WAND", "SPACE_MOUSE" or "WIIMOTE" (upper or lowercase).
  • DeviceURL: Device@Host[:Port]
  • AxisSpec: Maps XYZ tracker coordinates to world coordinates. Three character string comprised from 'XYZxyz' where upper-case indicates positive orientation, lower-case negative orientation. Ex., Xzy maps (X,Y,X) to (X,-Z,-Y)
  • Device: VRPN device name
  • Host: Host name or address
  • Port: Port number
  • Id: Id number
  • Matrix3f: [number x 9], a coordinate change matrix that transforms from tracker local coordinates to world coordinates.
  • *_Vector3f: [number, number, number]
The url field specifies the default URL to use for tracker, analog and button fields if their URL declaration is missing. Which fields are parsed depends on the device type. Gyration and Intersense devices require the tracker, button and analog fields; unless the url field is given, in which case the other ones can be omitted and default values will be used for any other parameter. The sensorid parameter defaults to "any" if not given. The parameters position_axis and attitude_axis default to the identity matrix.
For SpaceMouse and Wiimote you may specify analog or url, if both are omitted the device URL will default to "device@localhost" for SpaceMouse and "WiiMote@localhost" for Wiimote (actually these devices also require a button, but it's assumed to be the same URL than the analog device). Coherent devices could originate from different VRPN servers. Examples: 
 --vrpn-manipulator='{"type": "INTERSENSE_WAND", "url": "dtrackbody@z5-3-1:7702"}' 
 --vrpn-manipulator='{"type": "WIIMOTE", "analog" : { "url": "WiiMote@z5-3-1:7702"}}' 
 --vrpn-manipulator='{"type": "GYRATION_MOUSE",
                      "tracker": {"url": "dtrackbody@z5-fe",
                                  "sensorid": 1,
                                  "position_axis": [1, 0, 0,
                                                    0, 0,-1,
                                                    0,-1, 0],
                                  "attitude_axis": "XZY"},
                      "button": { "url": "Device0@z5-3-1:7701" },
                      "analog": "Device0@z5-3-1:7701"'}
Transparency options
 --alpha-blending Enable alpha blending for transparency. By default, multi-layer depth-peeling will be used.
 --slices value Number of slices of the depth partition used in multi-layer depth-peeling.
Frame capture
 --file-format File format to use for frame grabbing
 --file-prefix Name prefix to use for the files produced by frame grabbing. The prefix can include and absolute path.
 --frame-count frames Render the given number of frames after the command line scene has been created and then exit. This option has no effect if an interactive console is started. In combination with --idle-AA the number of output frames may be less than expected and no idle AA is performed.
 --grab-frames Use this option to write an image files per camera at 25fps, or per frame if a camera path is given.
Camera paths
 --path Set an RTNeuron specific camera path. This camera path must be in the format used by any other rendering engine from BlueHubCore. This option creates a manipulator that takes precedence over any other (e.g. VRPN manipulators).
 --path-fps float Sets the frames per second to use in camera path playback. If less or equal than 0 real time will be used.
Profiling
 --profile [option[,option]...] Enables profiling together with some additional profiling options. Valid option are:
  • channels=bool: True to enable profiling and logging of per channel draw/readback/assemble operations (False by default)
  • compositing=bool: False to disable frame compositing in multichannel configurations (True by default)
  • gldraw=bool: False to disable GL draw calls for neuron models (synapses and other objects are still drawn, True by default)
  • logfile=filename: File to write frame times in milliseconds ('./statistics.txt' by default')
  • readback=bool: False to disable frame readback and compositing in multichannel configurations (True by default)
When profiling is enabled the rendering loop will run without waiting for any event.
 --proof-frame filename If set, the first frame rendered after the scene has been loaded will be written to the given filename.
Equalizer options
 --eq-config config_file|session The .eqc config file or hw_sd session name for choose the configuration.
 --eq-layout layout Configuration layout to use
 --roi Enables region of interest clipping for framebuffer readback of Equalizer compounds.
 --window-size width height Window size hint to use if no Equalizer config is provided.

All RGB and RGBA tuples are space separated numbers between 0 and 1.
