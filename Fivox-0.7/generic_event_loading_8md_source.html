1 Generic event loading {#genericEventLoading}
2 =====================
3 
4 For research and debugging purposes, sometimes we need to quickly visualize
5 a set of events in 3D space. For example, in a neural circuit with synaptic
6 connectivity, we are interested in visualizing the position of the post-synaptic
7 cells; or for validation purposes, we want to check that the LFP computation is
8 done correctly for a predefined set of events.
9 
10 Since the variety of use cases is so wide, each of them with its own
11 particularities, we need to find a generic way of loading any predefined set of
12 events.
13 
14 The solution chosen for this feature is the loading of events from file. This
15 offers users flexibility to generate all types of events in a simple and generic
16 way, so the tool doesn't need to be adapted to fit the particular needs of any
17 situation.
18 
19 A generic event is always defined by its 3D position, radius and value,
20 regardless of the use case. It could be the cases that the radius and value are
21 not relevant, but they are always present (e.g. in the synaptic connectivity,
22 the radius can be 0, and value 1, as we will count each event as one cell).
23 
24 
25 ## Requirements
26 
27 * Users should be able to specify their own input events, as easily as possible,
28 from an external source (i.e. file).
29 * The format of the input file should be clearly defined and documented.
30 * All the library common capabilities should still be available for their use
31 with this feature. For example, generating 3D volumes with different
32 resolutions and using any available functor.
33 
34 
35 ## API
36 
37 A TestLoader is already present in Fivox, which generates a set of 7 events in a
38 fixed position, and is used for basic use cases such as the validation of the
39 LFP computation in a test case. The input URI in this case is as follows:
40 
41  "fivoxtest://?resolution=1&functor=lfp"
42 
43 What we need for the loading of any number of generic events is very similar to
44 this, but we need a way to specify the source of our events, instead of
45 generating them at runtime. We can use the BlueConfig path location as the event
46 file path. A change in the name might also be convenient, as it is not anymore
47 just for testing purposes. The loader and schema names are open for discussion.
48 
49  "fivox:///absolute/path/to/events/file?resolution=1&functor=lfp"
50 
51 The path to the input file is optional, behaving like the original TestLoader
52 and creating the initial 7 events if it is omitted. Please note that, when using
53 the GenericLoader, the events file path replaces the BlueConfig path (used
54 in most of the scientific use cases).
55 
56 For simplicity, and for now, only the GenericLoader can read events from file.
57 But any EventSource can write this type of files, dumping their contents (all
58 the events and their corrresponding values that it contains at the moment).
59 For that, a new method will be implemented in the base class:
60 
61  bool EventSource::write( const std::string& filename, bool binary = true );
62 
63 The corresponding option should also be added in the _voxelize_ application,
64 so users have access to it.

67 ## File format
68 
69 For the input file containing the events, two different formats should be
70 supported, binary for performance, and ASCII for readability, but keeping the
71 same layout.
72 
73 #### ASCII
74 
75 Header containing the version numbers and the format used, followed by all the
76 events (one event per line).
77 
78 Example:
79 
80 ```cpp
81 # Fivox events (3D position, radius and value), in the following format:
82 # posX posY posZ radius value
83 # File version: 1
84 # Fivox version: 0.6.0
85 Number of events: 10
86 50.115 1971 61.384 1.0 1
87 54.818 1997.1 71.462 1.0 2
88 50.896 1760.8 56.659 1.0 3
89 38.099 1807.9 30.819 1.0 4
90 19.888 1755.2 69.948 1.0 5
91 42.529 1730.5 50.998 1.0 6
92 59.509 1742.3 50.691 1.0 7
93 43.298 1826.1 57.443 1.0 8
94 52.034 1595.8 55.536 1.0 9
95 29.455 1451.9 28.081 1.0 10
96 ```
97 
98 #### Binary
99 
100 Two 32-bit values at the beginning (magic and version numbers), followed by
101 all the events, with five 32-bit floating point values
102 (posX posY posZ radius value) per event.

105 ## Issues
106 
107 ### 1: What names should we use for the loader and URI schema?
108 
109 Resolved: Use GenericLoader and "fivox://".
110 
111 We can replace the TestLoader by GenericLoader. Other alternatives are:
112 EventLoader, GenericEventLoader, FileLoader.
113 
114 As for the URI schema, we have "fivoxtest://", that could be replaced by
115 simply "fivox://" (which is now used for the CompartmentLoader, but that could
116 be changed), "fivoxfile://", "fivoxevents://", "fivoxgeneric://".
117 
118 ### 2: Why not extending this feature for any loader, to be able to store the event positions in a cache file after their initial creation?
119 
120 Resolved: Yes. We can write the events from any loader, but read only from
121 the GenericLoader (for now).
122 
123 It's an interesting possibility, specially considering the amount of time it
124 takes in some occasions to just create the events based on a BlueConfig file and
125 target, for example. The problem is that not every loader works the same way
126 (for example the SynapseLoader loads by chunks, not everything at once at
127 construction time), and this addition would also complicate the implementation
128 and usage of each of the loaders. For the events reading, we should keep this
129 feature self-contained in the new GenericLoader, at least for now, and then we
130 see if it is possible to extend it for any use case. For writing, there should
131 not be any problem in making it available for any loader.
