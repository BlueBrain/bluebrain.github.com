4 ## Motivation
5 
6 Redesign the current BBPSDK-inspired API for also allowing voltage traces.
7 The returned data should change to be a matrix[frame, compartment] instead
8 of the current vector[compartments].

15 * merge brion::CompartmentReport::loadFrame and loadNeuron in load( startTime, endTime )

17 * Change all timestamps to double

63 ## Issues
64 
65 ### 1: Do we need separate functions for frame-based and trace-based access?
66 
67 _Resolution: No_
68 
69 The current CRView::load( start, end ) API already allows selecting a single
70 Neuron and loading all frames. The brion API will need to change (see above).
71 
72 ### 2: Do we provide column-major memory layout?
73 
74 _Resolution: No_
75 
76 We will not reformat the data layout to be column-major, i.e., packing all
77 samples of a single compartment together. This is an optimization and we do
78 not expect today to load more than a few neurons for voltage traces.
79 
80 ### 3: Should we change the time stamps to double?
81 
82 _Resolution: Yes_
83 
84 Reasons: Not memory critical since it's not stored much. Python uses double too,
85 and with plasticity simulations this becomes critical. This will create some
86 intermediate inconsistency with other classes.
