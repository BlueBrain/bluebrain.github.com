1 Additions to compartmentReport to support ZeroEQ streams {#compartment_report}
2 ============
3 
4 ## Motivation
5 
6 The current CompartmentReport lacks an implementation based on ZeroEQ as the
7 SpikeReport does. There is a network plugin implementated with codash, but it
8 has been deprecated. Since there's a new specification of the SpikeReport API
9 ready for implementation, it makes sense to review the CompartmentReport
10 interface to make it consistent with the new SpikeReport and prepare it for the
11 ZeroEQ restrictions at the same time.
12 
13 This specification proposes a new interface for the CompartmentReport with
14 these goals in mind.

295 ## Implementation
296 
297 This section will cover some implementation details related to the stream
298 case as the file based IO is straight forward.
299 
300 The most two important decisions taken are:
301 - The granularity of the messages sent won't be the whole frame but a cell.
302 - The mapping will be always included as part of the frame.
303 
304 The rationale for these decisions follows.
305 
306 The current API for writing is already cell oriented, while this is convenient
307 to write on memory mapped files or H5 files without requiring the user to
308 assemble the whole buffer before writing, it's not consistent with the fact that
309 reads are whole frame oriented. This means that in the stream case we have to
310 assemble the frame at the writer or reader part of the implementation, or
311 change the API and force the user to do it on the writer side. Requiring
312 the user to assemble the frame is the easiest solution for the implementation
313 of the stream based case but it requires bigger changes to adapt existing code
314 both in the user side and in the implementation of existing plugins.
315 Therefore, the decision here is to keep the API similar to the current one
316 and send one message per cell. This allows interesting things like filtering
317 cell subsets on origin based on topics, something much more difficult to do
318 with whole frames. It also allows very easily to have multiple clients receiving
319 different portions of the circuit and only those. Obviuosly this introduces an
320 overhead, but we expect it to be small in relative and absolute terms. At the
321 same time we don't expect to stream full compartment data for detailed
322 morphologies for more than 10^5 cells at the moment. The implementation will
323 be revised if problems arise trying to do point to point streaming of larger
324 data sets. Having a separate message per cell also simplifies the creation of
325 the global mapping provided at the reader, because it can be created locally
326 based on the selected cell subset regardless of which cells are being
327 published by the writer. That said, soma only reports may receive special
328 treatment for efficiency reasons, but that probably implies an API extension
329 also.
330 
331 The reason for sending the mapping as part of every (per cell) frame is also
332 simplicity. If the mapping is a separate piece of information that needs to
333 be requested by clients, it adds complexity in several points. The first one
334 is the constructor, that instead of simply creating a subscriber, needs to
335 request the mapping synchronously. Changing the cell set of interest on the
336 reader is also more complicated, since it requires again a request that with
337 the current API must be synchronous. The proposed approach also avoids having
338 to add a finishMapping function to know then the mapping is complete and can
339 be sent to receivers potentially waiting in their constructors.
340 Related to the issue above we could distinguish between receiving a global
341 mapping or per cell mappings. If the mapping was requested separately, having
342 a whole frame, makes all the code more complicated. The overhead of including
343 the per cell mapping inside the data message can be estimated knowing the
344 maximum and average numbers of comparments section (unreported sections need
345 to be included in this calculation). For example, assuming at most 16
346 compartments per section and 4 compartments on average, the overhead is an
347 additional 1/32th of the frame size. In this case, for every 4 float numbers
348 (the compartment data) we have to add a compartment count that can be enconded
349 with 4 bits, in other words we have 4/(4*32) extra bits for each data bit. The
350 actual overhead still needs to be figured out, but anything around this 3%
351 seems reasonable, overall considering that is data size is a problem a big
352 reduction can be achived using compression of the floating point data.
353 
354 The messages of the protocol are:
355  - Publisher to subscribers:
356  * FRAME_DATA
357  * FRAME_BARRIER
358  * REPORT_METADATA
359  - Subscribers to publisher:
360  * REQUEST_METADATA
361 
362 The fields of a frame data message are the following:
363  - cell GID
364  - timestamp
365  - compartment data (n floats)
366  - compartment counts (m integers packed as 4 or 8 bit n = Sum counts)
367 
368 The timestamp is guaranteed to be the same for all cells in the same frame which
369 means that a frame can be considered finished when the last received timestamp
370 is different to the previous one. Nevertheless, in order to avoid receivers wait
371 until the first message of the next frame arrives, the FRAME_BARRIER message
372 is used to signal the end of a frame (this message is an empty event). The
373 FRAME_BARRIER message is used before the very first frame data message, this
374 is done because otherwise receivers do not have enough information to know
375 if the very first frame is a complete frame or they joined late.
376 
377 This protocol does not allows negative timestep in a stateless manner, bute
378 this can be easily addressed adding a timestep field to the message or a bit
379 to indicate the stream direction.
380 
381 The timestamp could be sent as a separate message emitted when the first cell
382 in a frame is sent, but this has the disadvantage of making the protocol
383 stateful and in case of traffic congestion the results would be incorrect if
384 that message is dropped.

390 ### 1: How does a the ZeroEQ implementation get the metadata from the subscriberat construction time?
391 
392 _Resolved: No_
393 
394 It will have to be by discovery of the data source via broadcasting, unless a
395 request/reply mechanism is implemented in ZeroEQ. The problem of this
396 broadcasting is that any publisher could answer while the report can be
397 created with a specific URI to connect to. This really looks like something
398 that should be implemented via request reply.
399 
400 ### 2: Subset selection should be performed at the receiver or sender side?
401 
402 _Resolved: No_
403 
404 This is undecided yet, but the specification of the API doesn't rely ony any
405 of the approaches and the protocol can support both.
406 
407 ### 3: Why selectCellSet() can make getCurrentTime() advance and miss data?
408 
409 _Resolved: No_
410 
411 The reason is simple, but it's a scenario that is exclusive of the stream case.
412 When frames are received they may have been already reduced to a given GID set
413 at origin or even locally. If selectCellSet requests a GID set which is not a
414 subset of the current one, the frames received but not yet consumed are not
415 valid for the new mapping, so they have to be thrown away. This efectively
416 makes the current time advance, even if the frames have not been returned.
417 
418 An alternative could be to make selectCellSet a completely asynchronous
419 request with a future, and then make the client responsible of checking if
420 the mapping has changed after every successful read operation. This alternative
421 can only be efficient if the report provides a hash of the mapping to avoid
422 expensive comparisons.
423 
424 Another solution is to remove this function and require creating a new reader
425 in order to change the mapping. This may be a good solution in the context of
426 the stream source, but maybe not be so good for files.
427 
428 ### 4: Why getCurrentTime returns the start time of the next read operation instead of the last one?
429 
430 _Resolved: Yes_
431 
432 For consistency with seek. If seek jumps to a point in the report, it seems
433 reasonable to make getCurrentTime return this value after completion. This is
434 only possible if getCurrentTime is referred to the next read operation. After
435 a read operation has been completed, the only thing needed to find out the
436 start time is to subtract the timestep.
437 
438 This is symmetric to SpikeReport::getCurrentTime(), the only disadvantage is
439 that the timestamp returned is not really part of the time window of the
440 the last frame read because [start, end) is open on the right. Admitedly, this
441 can be confusing, but the only reasonable alternative seems to be to return the
442 timestamp read along the data in readNextFrame.
