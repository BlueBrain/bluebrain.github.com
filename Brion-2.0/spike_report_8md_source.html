1 Redesign of the SpikeReport and Spikes container {#spike_reports}
2 ============
3 
4 The current design of the SpikeReport has several flaws that need to be
5 addressed:
6 * It doesn't allow partial loading for file based reports.
7 * It exposes too many details of the stream case which can probably be hidden.
8 * The container class is a std::multimap while it could simply be a sorted array
9  because no insertion is needed.
10 This specification proposes a new API to address these defficiencies as well
11 as include new requirements that need to be fulfilled.
12 
13 ## Requirements
14 
15 The high level requirements for the replacement are listed below. Note that
16 all these requirements do not need to be directly satisfied by
17 brion::SpikeReport, but it must be possible to do so by a higher level class
18 based on it.
19 
20 * Provide a container for spikes which is sorted by timestamp, storage
21  efficient, and random accessible at the <sup>n</sup>th position in amortized
22  constant time (provided the data has been already fetched). This is basically
23  a std::vector, but only read access is needed.
24 
25 * Allow partial data loading for file access.
26 
27 * Allow the selection of a subset of the cells present in the data source to be
28  reported. In stream-based reports this information should be made available
29  to the sender to allow filtering at the source. In the ZeroEQ implementation,
30  the same channel needed for communicating the desired target could be used
31  for the handshaking required to ensure that the first spikes are not missed.
32 
33 * As few details as possible that are specific to streams should be exposed.
34  This means that it should be easy to write client code that works the same
35  for both file based and stream based reports without special conditions for
36  stream based reports.
37 
38 * For stream reports the user will be able to:
39  * Retrieve all the data until the end of the stream in a simple loop without
40  doing any active wait (blocking calls are allowed).
41  * Know if the end of stream has been reached.
42  * Get all the data received so far without locking.
43  * Get information about the latest timestamp reached by the source in the
44  sender side even when there are no spikes at all (currently if the spike
45  report is completely empty it's not possible to make any progress until the
46  producer reaches the end of the report).
47 
48 * When data for a time interval is requested, the time intervals are always
49  closed on the left and open on the right.
50 
51 ## General remarks
52 
53 It has been agreed that unifying both the file and the stream cases under
54 a stream-like interface (in the sense of std::iostream or POSIX file descriptor
55 IO) seems to be the best abstraction at the lower level. Providing random
56 access and out-of-core support is complicated at the lower level because
57 internal buffering is needed and this can only be implementated efficiently if
58 some assumptions about how client code will use the API are made. The most
59 important characteristic of the current proposal is that read and write
60 operations can only move forward, so each operation starts where the previous
61 one left over. This way there is no need to make any distinction between the
62 file and stream based cases. Random access is partially supported by a
63 seek function that for stream based reports will only support forward skip.
64 
65 The current proposal defers important design decisions to the higher level API,
66 but the its benefit is that it provides a relatively simply API for the most
67 critical part.

102  * Release all resources.
103  * Any threads waiting on futures will return immediately and calling
104  * get will throw std::runtime_error.
105  */
106  ~SpikeReport();
107 
108  const URI& getURI();
109 
110  /** ? */
111  float getStartTime() const;
112  float getEndTime() const;
113 
114  /**
115  * @return the end time of the latest complete read/write operation or
116  * UNDEFINED_TIMESTAMP if no operation has been issued.
117  *
118  * Read operations are deemed as complete when the returned future is
119  * ready.
120  *
121  * The time inverval to which getCurrentTime refers is open on the
122  * right. That means that upon completion of a read or write operation
123  * no spike read or written may have a timestamp >= getCurrentTime().
124  */
125  float getCurrentTime() const;
126 
127  /**
128  * Read spikes until getCurrentTime becomes > min, the end of the report
129  * is reached or the report closed.
130  *
131  * This function fetches all the available data from the source.
132  * If no error occurs and the end is not reached, the min value passed
133  * is also guaranteed to be inside the time window of the data returned.
134  *
135  * Preconditions:
136  * - r.getState() is OK or ENDED.
137  * - The report was open in read mode.
138  * - There is no previous read or seek operation with a pending
139  * future.
140  * - min >= getCurrentTime() or UNDEFINED_TIMESTAMP.
141  *
142  * Postconditions:
143  * Let:
144  * - r be the SpikeReport
145  * - f be the returned future by r.read(min)
146  * - and s = r.getCurrentTime() before read is called (s = -inf if
147  * undefined):
148  * After f.wait() returns the following are all true:
149  * - r.getCurrentTime() >= s
150  * - If r.getState() == OK, then r.getCurrentTime() > min
151  * - For each spike timestamp t_s in f.get(),
152  * s <= t_s < r.getCurrentTime() (Note this could collapse to an
153  * empty interval if r.getState() != OK)
154  *
155  * After successful f.wait_for or f.wait_until, the result is the same
156  * as above. If they time out, no observable state changes.
157  *
158  * If the state is FAILED or some other operation is still pending
159  * when read is called the result is undefined.
160  *
161  * @param min The minimum end time until which spikes will be read. If
162  * UNDEFINED_TIMESTAMP, it will be considered as -infinite. This
163  * means that as much data as possible will be fetched without
164  * a minimum.
165  * @throw std::logic_error if one of the preconditions is not fulfilled.
166  */
167  boost::future< Spikes > read( float min );
168 
169  /**
170  * Read spikes until getCurrentTime() == max, the end of the report is
171  * reached or the report closed.
172  *
173  * This function is very similar to the normal read, but instead of
174  * specifying a lower bound of getCurrentTime at return, it specifies
175  * a requested strict value. The preconditions and postconditions
176  * are the same except for those involving the parameter min which
177  * become:
178  *
179  * Precondition: max > getCurrentTime()
180  * Postcondition: If r.getState() == OK, then r.getCurrentTime() == max
181  *
182  * @throw std::runtime_error if the precondition does not hold.
183  * @sa seek()
184  */
185  boost::future< Spikes > readUntil( float max );
186 
187  /**
188  * Seek to a given absolute timestamp.
189  *
190  * If toTimestamp >= getCurrentTime() and the report was opening
191  * for reading, data will be skipped forward until the timestamp
192  * is made current. In write mode for streams, consumers are notified
193  * about the progress.
194  *
195  * The case toTimestamp < getCurrentTime() is only supported by file
196  * based reports.
197  *
198  * Preconditions:
199  * - There is no standing read or readUntil operation.
200  *
201  * Postconditions:
202  * Let:
203  * - r be the SpikeReport
204  * - f be the returned future by r.seek(toTimestamp)
205  * Then:
206  * - After f.wait() returns r.getCurrentTime() == toTimestamp.
207  * - The postconditions of read operations imply that in forward skips
208  * this function throws away the data previous to toTimestamp (or
209  * avoids reading it at all if possible).
210  *
211  * @throw std::runtime_error if a precondition does not hold or the
212  * operation is not supported by the implementation.
213  */
214  boost::future< void > seek( float toTimestamp );
215 
216  enum State { OK = 0, ENDED = 1, FAILED = 2 };
217  /** @return The state after the last completed operation. */
218  State getState() const;
219 
220  /**
221  * Write the given spikes to the output.
222  *
223  * Upon return getCurrenTime() is the greatest of all the spike times
224  * plus an epsilon.
225  *
226  * @param spikes A sorted collection of spikes. For every spike, its
227  * timestamp t must be >= getCurrentTime().
228  */
229  void write( const Spikes& spikes );
230  }

234 
235 ### Copying a spike report from one URL to another without any timeout.
236 
237  SpikeReport input( "url1", brion::MODE_READ )
238  SpikeReport output( "url2", brion::MODE_WRITE )
239 
240  while( input.getState() == OK )
241  output.write( input.read().get() ))
242 
243  if (input .getState() == SpikeReport::FAILED)
244  ... // Report error.
245  // To be strictly correct, the last operation should be output.seek to
246  // advance until the current timestamp.
247 
248 ### Copying a spike report from one URL to another with a read timeout.
249 
250  SpikeReport intput( "url1", brion::MODE_READ )
251  SpikeReport output( "url2", brion::MODE_WRITE )
252 
253  while( input.getState() == OK )
254  {
255  auto future = input.read();
256  const auto status =
257  future.wait_for( boost::chrono::milliseconds( 250 ));
258  if( status == boost::future_state::ready )
259  output.write( future.get( ));
260  else
261  ... // Handle timeout
262 
263  output.write( spikes );
264  if (output.getState() == SpikeReport::FAILED)
265  ... // Report error.
266  }
267  if (input.getState() == SpikeReport::FAILED)
268  ... // Report error.
269 
270 ### Writing data at regular time intervals
271 
272  const float time delta = ...;
273  for( size_t i = 0; ...; ++i )
274  {
275  const start = i * delta;
276  const end = (i + 1) * delta;
277  const Spikes = produceSpikes( start, end );
278  output.write( spikes );
279  output.seek( end ).get();
280  }

287 ## Issues

289 
290 ### 1: Do we need a Spikes container?
291 
292 _Resolved: Yes_
293 
294 Not at this level. A higher level class can could provide a custom container
295 if that's necessary for performance reasons. At this class, the interface is
296 designed in a way that the implementation does not need to cache anything under
297 normal conditions (no operation times out). This means that the container
298 can be handed over to the consumer and rely on RVO and move semantics to
299 avoid unnecessary copies.
300 
301 ### 2: Why Spikes::getCurrentTime() returns a time greater than the largest spike time (in the general case)?
302 
303 _Resolved: Yes_
304 
305 To ensure consistency with the specification of the time interval from
306 SpikeReport::read, which is open on the right. The time of the last spike is
307 still easy to recover because it is the last element in the container.
308 
309 ### 3: The return value of read should be boost::future< Spikes& > or boost::future< Spikes >?
310 
311 _Resolved: No_
312 
313 It is desirable to avoid the reference and according to
314 [this reference](http://en.cppreference.com/w/cpp/thread/future/get), it seems
315 that get() returns a meaningful value only the first time is called when
316 returning by value (because the implementation uses std::move, but I'm not
317 fully sure about this).
318 
319 ### 4: Are the getStartTime() and getEndTime() methods needed?
320 
321 _Resolved: No_
322 
323 Seems convenient, but it's not clear what value should they return when the
324 data source is a stream and no data (or metadata) has been received yet.
