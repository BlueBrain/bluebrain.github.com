1 Redesign of the SpikeReport and Spikes container
2 ============
3 
4 The current design of the SpikeReport has several flaws that need to be
5 addressed:
6 * It doesn't allow partial loading for file based reports.
7 * It exposes too many details of the stream case which can probably be hidden.
8 * The container class is a std::multimap while it could simply be a sorted array
9  because no insertion is needed.
10 This specification proposes a new API to address these defficiencies as well
11 as include new requirements that need to be fulfilled.
12 
13 ## Requirements
14 
15 The high level requirements for the replacement are listed below. Note that
16 all these requirements do not need to be directly satisfied by
17 brion::SpikeReport, but it must be possible to do so by a higher level class
18 based on it.
19 
20 * Provide a container for spikes which is sorted by timestamp, storage
21  efficient, and random accessible at the <sup>n</sup>th position in amortized
22  constant time (provided the data has been already fetched). This is basically
23  a std::vector, but only read access is needed.
24 
25 * Allow partial data loading for file access.
26 
27 * Allow the selection of a subset of the cells present in the data source to be
28  reported. In stream-based reports this information should be made available
29  to the sender to allow filtering at the source. In the ZeroEQ implementation,
30  the same channel needed for communicating the desired target could be used
31  for the handshaking required to ensure that the first spikes are not missed.
32 
33 * As few details as possible that are specific to streams should be exposed.
34  This means that it should be easy to write client code that works the same
35  for both file based and stream based reports without special conditions for
36  stream based reports.
37 
38 * For stream reports the user will be able to:
39  * Retrieve all the data until the end of the stream in a simple loop without
40  doing any active wait (blocking calls are allowed).
41  * Know if the end of stream has been reached.
42  * Get all the data received so far without locking.
43  * Get information about the latest timestamp reached by the source in the
44  sender side even when there are no spikes at all (currently if the spike
45  report is completely empty it's not possible to make any progress until the
46  producer reaches the end of the report).
47 
48 * When data for a time interval is requested, the time intervals are always
49  closed on the left and open on the right.
50 
51 ## General remarks
52 
53 It has been agreed that unifying both the file and the stream cases under
54 a stream-like interface (in the sense of std::iostream or POSIX file descriptor
55 IO) seems to be the best abstraction at the lower level. Providing random
56 access and out-of-core support is complicated at the lower level because
57 internal buffering is needed and this can only be implementated efficiently if
58 some assumptions about how client code will use the API are made. The most
59 important characteristic of the current proposal is that all operations can only
60 move forward, so each operation starts where the previous one left over.
61 This way there is no need to make any distinction between the file and stream
62 based cases. Random access is partially supported by a function that does
63 forward skip.
64 
65 The current proposal defers important design decisions to the higher level API,
66 but the its benefit is that it provides a relatively simply API for the most
67 critical part.

230 ### Copying a spike report from one URL to another without any timeout.
231 
232  SpikeReport input( "url1", brion::MODE_READ )
233  SpikeReport output( "url2", brion::MODE_WRITE )
234 
235  while( input.getState() == OK )
236  output.write( input.read().get() ))
237 
238  if (input .getState() == SpikeReport::FAILED)
239  ... // Report error.
240  // To be strictly correct, the last operation should be output.forward to
241  // advance until the current timestamp.
242 
243 ### Copying a spike report from one URL to another with a read timeout.
244 
245  SpikeReport intput( "url1", brion::MODE_READ )
246  SpikeReport output( "url2", brion::MODE_WRITE )
247 
248  while( input.getState() == OK )
249  {
250  auto future = input.read();
251  const auto status =
252  future.wait_for( boost::chrono::milliseconds( 250 ));
253  if( status == boost::future_state::ready )
254  output.write( future.get( ));
255  else
256  ... // Handle timeout
257 
258  output.write( spikes );
259  if (output.getState() == SpikeReport::FAILED)
260  ... // Report error.
261  }
262  if (input.getState() == SpikeReport::FAILED)
263  ... // Report error.

285 ### 1: Do we need a Spikes container?
286 
287 _Resolved: Yes_
288 
289 Not at this level. A higher level class can could provide a custom container
290 if that's necessary for performance reasons. At this class, the interface is
291 designed in a way that the implementation does not need to cache anything under
292 normal conditions (no operation times out). This means that the container
293 can be handed over to the consumer and rely on RVO and move semantics to
294 avoid unnecessary copies.
295 
296 ### 2: Why Spikes::getCurrentTime() returns a time greater than the largest spike time (in the general case)?
297 
298 _Resolved: Yes_
299 
300 To ensure consistency with the specification of the time interval from
301 SpikeReport::read, which is open on the right. The time of the last spike is
302 still easy to recover because it is the last element in the container.
303 
304 ### 3: The return value of read should be boost::future< Spikes& > or boost::future< Spikes >?
305 
306 _Resolved: No_
307 
308 It is desirable to avoid the reference and according to
309 [this reference](http://en.cppreference.com/w/cpp/thread/future/get), it seems
310 that get() returns a meaningful value only the first time is called when
311 returning by value (because the implementation uses std::move, but I'm not
312 fully sure about this).
313 
314 ### 4: Are the getStartTime() and getEndTime() methods needed?
315 
316 _Resolved: No_
317 
318 Seems convenient, but it's not clear what value should they return when the
319 data source is a stream and no data (or metadata) has been received yet.
