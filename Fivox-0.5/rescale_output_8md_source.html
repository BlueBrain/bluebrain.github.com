1 Rescaling output voxel values
2 =============================
3 
4 Since Fivox makes all the computations in floating point values, and supports
5 different output types in the voxelize tool (just uint8 in the Livre
6 DataSource), we need a way to correctly rescale the resulting values, based
7 on the input data range and the output type range.
8 
9 ## Requirements
10 
11 * The internal computations should always be done in floating point values.
12 * The data source for Livre must always convert the resulting volume from
13  float to uint8_t values.
14 * The voxelize tool will always write a volume in the specified output data
15  type, converting it from float if needed.
16 * The process of rescaling the volume data should always consider the output
17  min and max values (known for each data type, e.g. 0-255 for uint8_t), and
18  the data range derived from the input (min and max values present after the
19  voxelization process in floating point values).
20 * The conversion should always be done at the end of the process, after all the
21  events' contribution has been considered to compute each voxel's value.
22 * The min and max values of the data range derived from the input
23  should be defined from the beginning, based on biological values and/or actual
24  data. Since this is very hard to know in advance, specially for time-dependent
25  data, a way to manually define this values should be provided. In any case,
26  there should be default values.
27 
28 ## Implementation
29 
30 Floating point values are already used along the whole process. And the
31 voxelize tool is already taking care of the rescaling when the output data type
32 is different than float.
33 
34 In the EventFunctor (base class for the functors) there is a _scale method,
35 used on each voxel to scale and clamp the value to the output data range. There
36 are several problems with the current implementation (see fieldFunctor, for
37 example):
38 
39 * _scale assumes that the value is always normalized between 0 and 1, which
40  might not be true, depending on the value of the 'magnitude' parameter.
41 * In the case of voltages, it's always shifting the event values by
42  brion::MINIMUM_VOLTAGE, to make all the values positive. This is wrong, as
43  soon as we start adding voltages together for the computation of one specific
44  voxel, we need to know their sign.
45 
46 The new implementation should get rid of these shift and scale operations in the
47 functors, and use always the raw values of the report (or the result of the
48 corresponding operation for an specific use case, e.g. LFP). All the conversion
49 operations should be done only at the end of the process, once we make sure that
50 the conversion won't affect the result of the computation.
51 
52 The 'magnitude' parameter in the input URI is not enough for a correct rescaling
53 operation. It was added with the intention to hold the value for the input data
54 range, but this assumes that the starting value is 0, incorrect as soon as we
55 start dealing with summations of voltages with different signs, for example.
56 
57 So two new parameters should be added, 'inputMin' and 'inputMax'. And
58 potentially remove the 'magnitude'. Example:
59 
60  voxelize --volume fivoxlfp:///path/to/BlueConfig?report=currentReport&target=targetName&inputMin=-50&inputMax=100
61 
62 The operation to be performed in order to rescale a value from the input data
63 range to an output data type range is as follows:
64 
65  outputValue = (inputValue - inputMin) * (outputMax - outputMin) / (inputMax - inputMin) + outputMin
66 
67 ## Issues
68 
69 ### 1: Rescale on the full volume or in the functor (on each voxel)?
70 
71 Resolved: Rescale the full volume at once, after it's been generated.
72 
73 We can do this operation at the end of the functor, once the summation has been
74 done for a specific voxel, but we can also do it at the very end, once the
75 full volume has been generated. ITK provides a filter that does this for us
76 (IntensityWindowingImageFilter), so we can make use of it easily, adding
77 this rescale filter as an additional step between the generation of the float
78 volume and its final use (writing in the case of the voxelize tool, or passing
79 it to Livre in the case of the data source).
80 
81 Update: Rescale in the functor (on each voxel).
82 
83 After a closer review, we noticed the performance implications of using a
84 FloatVolume in the Livre data source. The only acceptable solution in that case
85 is to use a ByteVolume from the beginning, so the scaling needs to be done
86 in the functors. In that case, the ITK filter mentioned above would only be used
87 in the voxelize app, and the new solution would implement the same behavior
88 inside the EventFunctor class, deriving all the functors from it.
