1 Modification of the brion::Morphology read API {#morphologyRead}
2 ============
3 
4 ## Motivation
5 
6 The current read API is incorrect. It does return a shared_ptr to mutable data,
7 but is marked const. For example, the brain::Morphology takes a const
8 brion::Morphology&, but the given object is modified in case of SWC with global
9 transformation.
10 
11 ## API variants:
12 
13 We chose option 5 since it clarifies and unifies the behaviour of all plugins
14 with a clean API:
15 
16 1. ConstVectorPtr readFoo() const
17 2. Vector readFoo() const
18 3. VectorPtr readFoo()
19 4. std::future<Vector> readFoo() const
20 5. Vector& getFoo() and const Vector& getFoo() const
21 
22 Some implementations do read multiple vectors at the same time (SWC, ZeroEQ).
23 
24 Option 1 is rejected: Client code always needs to make a copy if it wants to
25 modify the data.
26 
27 Option 2 either complicates the implementation of SWC/ZeroEQ which need to
28 reread data if a given read function is called multiple times, or would
29 constrain the API to throw if a read function is called multiple times.
30 
31 Option 3 is awkward since it is a mutable read API, and delays the problem to
32 the user of the API.
33 
34 Option 4 adds the future complexity (with shared state between readFoo/readBar)
35 on top of Option 2.
36 
37 Option 5 parses always the data in the ctor using a future and thread pool
38 synchronized in all getFoo() calls. This forces that the brion::Morphology has
39 state and data, and we should remove the write API at the same time. It has the
40 disadvantage of loading all data deferred from the ctor, which requires out of
41 core loading (or batching) for large amounts of morphologies. Client code using
42 the brain API already has this constraint. While this seems to be more complex,
43 actual plugin implementations become simpler since all of the threading and
44 synchronization will be done in the plugin base class.
